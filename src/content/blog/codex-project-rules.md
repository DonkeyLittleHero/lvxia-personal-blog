一段很长的提示词不一定更有效。对编码任务来说，最重要的是让 Codex 知道结果是什么、哪些东西不能碰、怎样证明工作完成。

## 先写结果，不要先写工具

与其说“请用某某框架做”，不如先说“我需要一个可以直接部署到 GitHub Pages、手机与电脑都可用的个人博客”。

工具会影响实现，但结果决定方向。只有在技术选择确实是约束时，才把框架写死。

## 一份最小任务说明包含六部分

```text
目标：最终用户能完成什么
现状：已有文件、技术栈和已知问题
范围：这次必须完成的页面或功能
边界：禁止修改、禁止上传或不能新增的部分
验收：需要运行哪些命令、检查哪些场景
交付：希望得到代码、说明、截图还是发布链接
```

这六项比“写得非常详细”更重要，因为它们分别解决方向、上下文、优先级、安全、质量与交接问题。

## 把审美要求变成可观察的描述

“做得炫酷”很难验收。可以写成：深色背景、鼠标移动时极光产生轻微视差、卡片有玻璃折射和三维倾斜、动画在减少动态偏好下关闭。

这样既保留创意空间，也能检查是否完成。

## 让 Codex 自己验证

说明中应明确要求安装依赖、运行构建、检查移动端和桌面端，并修复发现的问题。对于网站项目，还可以要求检查所有站内链接和 404 页面。

## 一个可直接复用的结尾

```text
请直接完成实现，不要只给方案。完成后运行构建和必要测试，
修复所有由本次修改引入的问题，并用简短清单说明改了什么、
验证了什么，以及我下一步如何发布。
```

好的任务说明不是控制每一行代码，而是把交付契约写清楚。
